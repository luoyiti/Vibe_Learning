# 知识沉淀系统 · 项目背景与设计说明（CONTEXT）

> 这份文档给即将构建本项目的 AI agent 提供完整设计背景：**是什么、为什么这么设计、要遵守哪些不变量**。
> 实现指令在配套的 `BUILD_PROMPT.md`。本文解释"为什么"——请在动手前通读，避免实现时把关键设计当成可优化项给改掉。

---

## 1. 项目是什么

一个把"与 AI 对话中学到的知识"沉淀为可复用资产的个人学习系统。

向任意 AI agent（Claude Code / Codex / Cursor / 对话类皆可）提出学习问题，agent 不受模板约束、自由选择最合适的方式讲解（notebook、单文件 HTML、React、图或纯文字）。但每一次学习都有一条绕不过的**硬性要求——有产出、有沉淀**，它同时包含两样输出：

1. **自由产物**：一个自包含的东西（脱离当下对话、日后单独打开也能看懂）。
2. **格式化信封**：一份元数据（一句话总结 + 标签 + 关系 + 元信息）。

这些沉淀统一落入存储层；其中**文件是事实来源**，承载产物与正文，而索引／知识图谱只是从文件派生、随时可重建的查询层。

**核心循环**：提问 → agent 自由讲解 → 双产出 → 落入存储 → 派生索引（与未来的知识图谱） → 按需复盘内化。

---

## 2. 设计原则（不变量 — 不可违背）

这四条是地基，实现时不得为图方便而违背。

1. **文件即事实来源。** 索引与知识图谱全部从单元派生、可随时重建。检验铁律：删掉派生层，系统能否仅凭单元（`notes/`）完整重生。
2. **判断与执行分离。** LLM agent 只做判断／生成／分析（不确定）；确定性脚本（工具层）负责全部读写／构建。**agent 永不直接触碰文件系统**，所有 I/O 经工具层 CLI。这是"实施不易出错"的根本——把易错的机械工作放进可测脚本，agent 只下判断。
3. **机器严格 / 人类自由。** 承重墙只设在 `meta.json`（严格 JSON Schema、工具校验）；正文与产物完全自由，不被结构污染。
4. **ID 即身份。** 单元以稳定 **ULID** 标识；索引、图谱、关系一律引用 ID 而非路径，改名移动不破坏任何引用。

---

## 3. 架构：四层 / 三 agent

自上而下四层堆叠，**写入沿层向下、读取沿层向上**，层间不跨越——所有文件 I/O 收口在工具层。

| 层 | 职责 |
|---|---|
| ① Agent 层 | 三个由提示词驱动的 LLM；负责判断／生成／分析，不直接读写文件。 |
| ② 工具层 | 不可修改的确定性脚本（CLI）；承担全部读写与构建，是触碰存储的唯一通道。 |
| ③ 存储层 | 统一知识单元（文件）是唯一事实来源；索引与知识图谱从单元派生、可重建。 |
| ④ 展示层 | 只读前端（已实现：`ks serve` 本地 web UI）；检索筛选、翻阅单元、打开产物与原始对话。图谱渲染待 `graph build` 落地后接入。 |

**三个 agent 的职责**

- **学习指导 agent**：回答学习问题，自由讲解，但必须产出"自由产物 + 草稿单元"。
- **内容持久化 agent**：系统的**唯一写入者**。把学习产出抽象成合规的统一知识单元、纠错、维护索引。分配关系前会先**查询**索引找相关单元（read‑modify‑write）。
- **全局分析 agent**：**只读**。按索引与时间复盘学习内容，产出复盘报告。

**核心不变量**：所有文件 I/O 仅经工具层；写入只走「内容持久化 → 工具」**一条路径（单写者）**，store 的一致性只需守住这一条。

> 注：复盘**不**做成带调度状态（间隔、卡片、记忆因子）的独立复习子系统——那会引入大量可变状态、徒增复杂度。复盘改由全局分析 agent 按索引与时间**按需**完成，无状态。

## 4. 统一知识单元

全系统唯一的内容抽象。**一次学习 = 一个目录**（因 React 等富产物可能是多文件）。目录内只放事实来源；派生的索引与图谱在存储层另置。

```
20260620-143000--db-isolation-levels--01HXY8ZQ3F.../   目录名 = 时间戳--slug--短ID
├── meta.json     ★ 元数据信封（承重墙）：严格 schema、索引与图谱的唯一依据
├── note.md       ★ 规范文本正文：知识的耐久文本，检索/图谱/复盘都读它
├── artifact/     ○ 自由产物（富表现形式，自包含、可独立打开）
│   └── index.html    或 notebook.ipynb / React 构建 / diagram.svg
├── assets/       ○ note.md 引用的图片、数据
└── source/       ○ 溯源留档（原始对话；original_question 已结构化进 meta）
```

三个组成：

- **`meta.json`（★ 必需）**：严格 JSON、工具校验的承重墙；索引与图谱完全据此构建。字段精确定义见 `schema/unit.schema.json`（由构建产出）。
- **`note.md`（★ 必需）**：知识的耐久文本表示。工具只能读文本，富产物（React/notebook）未必可提取，故**文本正文是地基**，也是富产物"腐烂"后的耐久兜底。
- **`artifact/`（○ 可选）**：可交互／可视的富产物。纯文本学习可没有它，此时 `note.md` 即产物（`meta.artifact.format = "markdown"`，`entry = "note.md"`）。

**`meta.json` 字段（按用途分组）**

- *身份与版本*：`id`(ULID)、`schema_version`、`title`、`slug`
- *信封核心*：`summary`(一句话)、`original_question`(触发学习的问题，最准的检索钩子)、`tags`(归一化标签)
- *关系 · 图谱来源*：`concepts`(概念节点，多单元借共享概念连通)、`relations`(有类型的单元边：prerequisite/extends/related/contrasts/applies/part_of)
- *产物描述符*：`artifact`{ format, entry, self_contained }
- *来源与时间*：`created_at`、`updated_at`、`source`{ channel, model, transcript }
- *工作流与可选*：`status`(draft/stable/needs_review)、`references`、`confidence`

## 5. 关键设计决策（为什么——别把它们改掉）

- **不设独立复习子系统**：间隔重复需要可变调度状态，是设计中最重的部分。改由全局分析 agent 按索引与时间"复盘"，无状态。
- **知识图谱来自声明式关系**：关系判断由能纵览全库的内容持久化 agent 写入单元元数据（`concepts`/`relations`），图谱抽取脚本只做**确定性装配**。判断与装配分离，化解了"无状态 agent 无法可靠建链接"。
- **`meta.json` 用独立 JSON 而非 YAML frontmatter**：JSON 无歧义、易做 schema 校验、agent 更不易写错。与正文分两文件，由持久化 agent 统一维护、不会失同步。
- **ULID 而非 UUID**：可按生成时间字典序排列，文件名前缀排序与之一致，既稳定又自带时间序。
- **`note.md` 保证一份可提取文本**：检索／图谱／复盘等工具只能读文本，文本正文既是工具输入地基，也是耐久兜底。

## 6. 当前阶段范围（重要）

本次只构建 MVP。明确的边界见 `BUILD_PROMPT.md`，这里给出意图：

- **现在构建（真实实现）**：存储层（目录约定 + Schema）、工具层的 `ingest`/`validate`、`reindex`、`query`，以及三个 agent 契约文件、仓库脚手架与最小测试。展示层其后已落地为 `ks serve`（只读 web UI，数据仅来自索引/查询层，见 `display/README.md`）。
- **占位（只留接口、不实现）**：知识图谱抽取工具（`graph build`）；展示层的图谱视图随之待接入。
- **数据接口现在就保留**：`meta.json` 的 `concepts`/`relations` 字段**从第一天起就采集**。这样未来图谱工具实现时，**所有历史单元已自带数据**。
- **解耦红线**：索引与查询**绝不依赖**图谱；图谱是纯增量，缺它系统照常工作。三个 agent 现在都能正常运转（全局分析用索引复盘、不需要图谱）。
