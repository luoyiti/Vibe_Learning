# OLAP 与 OLTP 的区别

**一句话：** OLTP 支撑「正在发生的业务事务」（快、准、强一致）；OLAP 支撑「对历史数据的分析决策」（重查询、大扫描、常接受延迟）。

## 1. 定义

| | OLTP | OLAP |
|---|---|---|
| 全称 | Online Transaction Processing | Online Analytical Processing |
| 目标 | 执行业务操作 | 提炼数据洞察 |
| 典型操作 | INSERT / UPDATE / DELETE + 点查 | SELECT 聚合、多维分析 |
| 数据模型 | 规范化（3NF） | 星型/雪花、宽表、反规范化 |
| 存储倾向 | 行存 | 列存 |
| 一致性 | ACID 强一致 | 常最终一致 / T+1 |

## 2. 工作负载差异

**OLTP**：高并发、短事务、每次触及少量行。用户是前台应用和终端。失败代价是资金损失与业务中断。

**OLAP**：并发较低，单查询可能扫描海量历史数据。用户是分析师与 BI 工具。优化方向是分区、列存、物化视图、向量化执行。

## 3. 为何常分开部署

分析查询直接打 OLTP 库会导致锁竞争与缓存污染（「分析拖垮业务库」）。标准路径：

```
OLTP → ETL/ELT（CDC、批处理）→ 数仓/OLAP → BI 报表
```

**HTAP**（TiDB、SingleStore 等）尝试在同一集群混合事务与分析，适合部分中小规模场景；大规模仍倾向专用分工。

## 4. 实例（电商）

- **OLTP**：支付时扣库存、写订单——几行数据，必须事务原子。
- **OLAP**：上月各品类 GMV 同比——百万行聚合。
- **OLTP+流**：实时风控拦截——毫秒级单笔查询。
- **OLAP**：推荐模型训练——导出行为宽表离线批处理。

## 5. 易错点

1. 「只有一个 PostgreSQL」不等于没有 OLAP 负载——复杂报表仍是分析型，只是缺少专用优化。
2. OLAP 数据不一定旧——CDC + 流可把延迟压到分钟级，但一致性弱于 OLTP 事务。
3. 列存 ≠ OLAP 定义本身，只是常见实现手段。
4. 数仓是架构角色，OLAP 是处理类型——二者相关但不等同。

**选型：** 强事务 + 频繁单行更新 → OLTP；跨历史大数据聚合 → OLAP/数仓；两者都需要 → 分离同步或评估 HTAP 边界。
