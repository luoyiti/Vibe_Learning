OLAP 与 OLTP：
两类在线数据处理范式

同一组织往往同时需要「把业务跑起来」和「把数据看清楚」——这两件事对数据库的要求截然不同。

Online Transaction Processing · Online Analytical Processing

摘要

Abstract OLTP（联机事务处理）面向日常业务操作：大量短小、高并发的增删改查，强调 ACID 与低延迟。 OLAP（联机分析处理）面向决策分析：复杂只读查询、跨维度聚合与历史趋势，强调吞吐与扫描效率。 实践中二者常通过 ETL/ELT 管道分离部署，而非在同一套 schema 上混用。

1. 定义与核心诉求

OLTP

联机事务处理——支撑正在发生的业务：下单、转账、挂号、库存扣减。每次操作通常只触及少量行，要求毫秒级响应与强一致性。

OLAP

联机分析处理——支撑事后决策：销售额趋势、用户留存、区域对比。查询往往扫描大量历史数据，做 SUM / GROUP BY / JOIN 多维聚合。

OLTP 的典型特征

  • 工作负载：高并发、短事务；读写比接近 1:1 或写多读少
  • 数据模型：高度规范化（通常 3NF），减少冗余、保证更新一致
  • 查询模式：按主键 / 索引点查，INSERT、UPDATE、DELETE
  • 一致性：ACID 事务是默认要求（银行余额不能「算错」）
  • 存储倾向：行存（row-oriented）——一次读/写整行记录更高效

OLAP 的典型特征

  • 工作负载：并发用户较少，但单个查询重、耗时长
  • 数据模型：反规范化——星型 / 雪花 schema，宽表、预聚合
  • 查询模式：只读为主；全表扫描、多维切片（slice / dice / drill-down）
  • 一致性：常接受最终一致或 T+1 延迟（分析不必与每一笔交易实时同步）
  • 存储倾向：列存（column-oriented）——只读少数列时 I/O 更少

记忆口诀：OLTP 关心「这一笔对不对、快不快」；OLAP 关心「过去一段时间整体上发生了什么」。

3. 典型架构：为何常分开部署

把分析查询直接打到 OLTP 库上，容易导致锁竞争、缓存失效、索引失效——俗称「分析拖垮业务库」。成熟做法是把数据从 OLTP 抽取、转换、加载到专用分析层。

HTAP：试图二合一

部分新一代数据库（如 TiDB、SingleStore、Snowflake 的混合工作负载）宣称 HTAP（Hybrid Transaction/Analytical Processing）——同一集群同时扛事务与分析。这在中小规模或强实时分析场景有价值，但大规模下「专用 OLTP + 专用 OLAP」仍是主流分工。

5. 易错点

误区 1：「我们只有一个 PostgreSQL，所以只有 OLTP。」——在 PG 上跑复杂报表仍是分析负载；只是没有专用 OLAP 优化，容易变慢或影响业务。
  • 误区 2：OLAP 数据一定比 OLTP 旧。近实时数仓（CDC + 流处理）可把延迟压到分钟级；但一致性语义通常弱于 OLTP 事务。
  • 误区 3：列存 = OLAP，行存 = OLTP。存储格式是重要实现手段，但不是定义本身；也有行存分析库、列存事务库（带局限）。
  • 误区 4：数据仓库就是 OLAP。数仓是架构角色；OLAP 是处理类型。数仓里的查询大多是 OLAP，但 OLAP 也可发生在 lakehouse、嵌入式 DuckDB 等形态。

选型简表

  • 需要强事务、频繁单行更新 → OLTP 库
  • 需要跨大量历史数据聚合、BI → OLAP / 数仓
  • 两者都需要 → 分离 + 同步管道，或评估 HTAP 产品边界
