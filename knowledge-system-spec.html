知识沉淀系统 Knowledge Sedimentation System · Design Spec

把与 AI 对话中学到的知识,沉淀为可复用、可复盘的结构化资产。

向任意 AI agent 提出学习问题,agent 不受模板约束、自由选择最合适的方式讲解;但每一次学习都有一条绕不过的硬性要求——有产出、有沉淀。它同时交付两类输出:自由产物(自包含,脱离当下对话日后也能单独看懂)与 格式化信封(一句话总结、标签、关系、元数据)。这些沉淀统一落入存储层,其中文件是事实来源,索引与知识图谱只是从文件派生、随时可重建的查询层。

★ 文件 · 事实来源 ☆ 派生 · 可重建 Agent 层 工具层 存储层 展示层
01

项目概述

Overview

系统把"学习"标准化成一条可重复的流水线:提问 → agent 自由讲解 → 双产出(产物+信封) → 落入存储 → 派生索引与知识图谱 → 按需复盘内化。agent 的讲解方式完全自由(notebook、单文件 HTML、React、图或纯文字皆可),被约束的只有"必须产出可沉淀的东西"这一点。

学习指导 agent 生成的每一份内容,都会被抽象成一个统一知识单元——这是全系统唯一的内容抽象。围绕它,内容持久化 agent 负责维护(纠错、维护索引与图谱),全局分析 agent 负责按索引与时间复盘。

范围说明 复盘不做成带调度状态(间隔、卡片、记忆因子)的独立复习子系统——那会引入大量可变状态、徒增复杂度。复盘改由全局分析 agent 按索引与时间戳按需完成:无状态、无机械负担。
02

设计原则

Invariants

01 文件即事实来源

索引与知识图谱全部从单元派生、可随时重建。检验铁律:删掉派生层,系统能否仅凭单元完整重生。

02 判断与执行分离

LLM agent 只做判断/生成/分析(不确定);确定性脚本负责全部读写/构建。agent 永不直接触碰文件系统。

03 机器严格 / 人类自由

承重墙只设在 meta.json(严格 schema、工具校验);正文与产物完全自由,不被结构污染。

04 ID 即身份

单元以稳定 ULID 标识;索引、图谱、关系一律引用 ID 而非路径,改名移动不破坏任何引用。

03

系统架构

Layered Architecture

四层自上而下堆叠。写入沿层向下(Agent → 工具 → 存储),读取沿层向上;层与层之间不跨越——所有文件 I/O 都收口在工具层。

层职责
Agent 层三个由提示词驱动的 LLM,负责判断、生成与分析;不直接读写文件。
工具层不可修改的确定性脚本,承担全部读写与构建,是触碰存储的唯一通道。
存储层统一知识单元(文件)是唯一事实来源;索引与知识图谱从单元派生、可重建。
展示层只读前端,渲染知识图谱、提供检索与筛选。
核心不变量 所有文件 I/O 仅经工具层;写入只走「内容持久化 → 工具」一条路径(单写者),store 的一致性只需守住这一条。这条分工(LLM 管判断、脚本管 I/O)正是"实施不易出错"的根本保障。
04

核心数据流

Data Flow

把双向流拆成两条互不交叉的路径:沉淀(写)与消费(读)。

沉淀路径 · 写入
用户输入→ 学习指导〔生成产物〕→ 内容持久化〔抽象成单元〕→ 单元写入·校验→ ★ 统一知识单元→ 索引构建 · 图谱抽取→ ☆ 索引＋☆ 知识图谱
消费路径 · 读取
☆ 索引 / 单元→ 查询·复盘取数→ 全局分析→ 复盘报告
☆ 知识图谱→ 展示层→ 用户浏览
05

统一知识单元

The Unit

一次学习 = 一个目录(因 React 等富产物可能是多文件)。目录内只放事实来源;派生的索引与图谱在存储层另置。

目录结构

20260620-143000--db-isolation-levels--01HXY8ZQ3F.../   目录名 = 时间戳--slug--短ID
├── meta.json     ★ 元数据信封(承重墙):严格 schema、索引与图谱的唯一依据
├── note.md       ★ 规范文本正文:知识的耐久文本,检索/图谱/复盘都读它
├── artifact/     ○ 自由产物(富表现形式,自包含、可独立打开)
│   └── index.html    或 notebook.ipynb / React 构建 / diagram.svg
├── assets/       ○ note.md 引用的图片、数据
└── source/       ○ 溯源留档(原始对话;original_question 已结构化进 meta)
★meta.json——严格 JSON、工具校验的承重墙;索引与图谱完全据此构建。
★note.md——知识的耐久文本表示。工具只能读文本,富产物未必可提取,故文本正文是地基。
○artifact/——可交互/可视的富产物;纯文本学习可没有它,此时 note.md 即产物。

meta.json 字段(按用途分组)

身份与版本
id(ULID,稳定主键)· schema_version · title · slug
信封核心
summary(一句话总结)· original_question(触发学习的问题,最准的检索钩子)· tags(归一化标签)
关系 · 图谱来源
concepts(概念节点,多单元借共享概念连通)· relations(有类型的单元边:前置/深化/相关/对比/应用/从属)
产物描述符
artifact:{ format · entry · self_contained } —— 告诉展示层和"重新打开"如何开启
来源与时间
created_at · updated_at(复盘按时间分析)· source:{ channel · model · transcript }
工作流与可选
status(draft/stable/needs_review)· references(外部引用)· confidence(一次性自评,仅供复盘分组)

强约束契约 · JSON Schema(写入工具据此校验,additionalProperties:false 拒绝未知字段)

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "additionalProperties": false,
  "required": ["schema_version","id","title","summary","original_question","tags","artifact","status","created_at","updated_at"],
  "properties": {
    "schema_version": { "type":"string", "pattern":"^\\d+\\.\\d+$" },
    "id":    { "type":"string", "pattern":"^[0-9A-HJKMNP-TV-Z]{26}$" },   // ULID,大写 Crockford base32
    "title": { "type":"string", "minLength":1, "maxLength":80 },
    "slug":  { "type":"string", "pattern":"^[a-z0-9-]+$" },
    "summary":           { "type":"string", "minLength":1, "maxLength":200 },
    "original_question": { "type":"string", "minLength":1 },
    "domain": { "type":"string" },
    "tags":     { "type":"array", "minItems":1, "uniqueItems":true,
                  "items":{ "type":"string", "pattern":"^[a-z0-9-]+$" } },
    "concepts": { "type":"array", "uniqueItems":true, "items":{ "type":"string", "minLength":1 } },
    "relations":{ "type":"array", "items":{ "$ref":"#/$defs/relation" } },
    "references":{ "type":"array", "items":{
        "type":"object", "additionalProperties":false, "required":["url"],
        "properties":{ "title":{"type":"string"}, "url":{"type":"string","format":"uri"} } } },
    "artifact": {
      "type":"object", "additionalProperties":false, "required":["format","entry"],
      "properties":{
        "format":{ "enum":["markdown","html","notebook","react","svg","image","none"] },
        "entry":{ "type":"string", "minLength":1 },
        "self_contained":{ "type":"boolean" } } },
    "source": { "type":"object", "additionalProperties":false,
                "properties":{ "channel":{"type":"string"}, "model":{"type":"string"}, "transcript":{"type":"string"} } },
    "status":     { "enum":["draft","stable","needs_review"] },
    "confidence": { "enum":["shaky","moderate","solid"] },
    "created_at": { "type":"string", "format":"date-time" },
    "updated_at": { "type":"string", "format":"date-time" }
  },
  "$defs": {
    "relation": {
      "type":"object", "additionalProperties":false, "required":["type","target"],
      "properties":{
        "type":{ "enum":["prerequisite","extends","related","contrasts","applies","part_of"] },
        "target":{ "type":"string", "pattern":"^[0-9A-HJKMNP-TV-Z]{26}$" },
        "note":{ "type":"string" } } }
  }
}

字段 → 派生消费者(验证设计完整性)

派生消费者读取的字段
索引构建id · title · summary · tags · domain · concepts · created_at · status ＋ note.md 全文
图谱抽取concepts(概念节点) · relations(有类型的单元边)——纯确定性装配
全局分析 · 复盘created_at · updated_at · tags · domain · confidence

三者所需字段全在 meta.json 内,故"删掉索引与图谱能否从单元完整重生"这条铁律成立。

06

关键设计决策

Key Decisions · Rationale

不设独立复习子系统

间隔重复需要可变的调度状态(间隔、卡片、记忆因子),是上一版最重的部分。改由全局分析 agent 按索引与时间"复盘",无状态,复杂度骤降。

知识图谱来自声明式关系

关系判断由能纵览全库的内容持久化 agent 写入单元元数据,图谱抽取脚本只做确定性装配。判断与装配分离,化解了"无状态 agent 无法可靠建链接"的难题。

meta.json 用独立 JSON 而非 YAML frontmatter

JSON 无歧义、易做 schema 校验,agent 也更不易写错(避免 YAML 缩进坑)。代价是与正文分两个文件,但由持久化 agent 统一维护,不会失同步。

ULID 而非 UUID

ULID 可按生成时间字典序排列,文件名前缀排序与之一致,既稳定又自带时间序。

note.md 保证一份可提取文本

检索、图谱、复盘等工具只能读文本;React/notebook 等富产物未必可提取。文本正文既是工具的输入地基,也是富产物"腐烂"后的耐久兜底。

待定与下一步

Open Items

以下为尚需敲定、提请审阅的事项。

工具层四脚本的 I/O 契约

写入·校验 / 索引构建 / 图谱抽取 / 查询取数 各自的输入、输出,以及对 meta.json 的读写边界——定清即可开工。

采集落盘按渠道区分

Claude Code / Cursor 可直接写盘;claude.ai 等对话类碰不到文件系统,需一个低摩擦的 CLI 导入(吃粘贴的产物+信封,校验后归档)。这是系统能否存活的关键。

标签受控词表与归一化

避免 ML / machine-learning / 机器学习 漂移;现在就定一份词表 + 落盘时归一,免得攒到数百条再治理。

整库进 git

版本史、备份、同步、diff 全部白送,并强化"文件即真相";知识被反复修订时,纠错与回溯都有迹可循。
